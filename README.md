# Financial Transactions Data Warehouse — ETL & Star Schema

An end-to-end ETL pipeline that ingests raw, messy synthetic fintech transaction data and builds a proper **star schema data warehouse** from it — one fact table surrounded by conformed dimension tables. Built to demonstrate data warehousing and modeling fundamentals for a Data Engineer role, with a financial-services dataset by design.

All data is **synthetically generated** (`etl/pipeline.py`) — not real financial data from any institution.

## Why this project

Built specifically to demonstrate two things a general analytics portfolio doesn't usually cover: **hands-on star schema data modeling** and **staged ETL pipeline design** (extract → transform → load as discrete, logged steps) — the same pattern orchestration tools like Airflow schedule as a DAG.

## Honest scope note

This project does **not** use Airflow, dbt, or a cloud warehouse (Snowflake/BigQuery/Redshift). It uses **SQLite as a local stand-in warehouse** and a plain Python script to run the ETL stages sequentially. It demonstrates the underlying data modeling and pipeline logic those tools would orchestrate in production — not hands-on experience with the tools themselves.

## Star Schema Design

```
                    dim_date
                        |
dim_customer ---- fact_transactions ---- dim_merchant
                        |
                  dim_account
```

**Fact table:** `fact_transactions` — one row per transaction, foreign keys to all four dimensions, `amount` and `status` as measures.

**Dimensions:**
- `dim_customer` — customer_sk, city, segment, signup_date
- `dim_account` — account_sk, account_type, opened_date
- `dim_merchant` — merchant_sk, merchant_name, category
- `dim_date` — standard date dimension (year, month, day_of_week, is_weekend)

All dimensions use surrogate keys (`_sk`), generated during the transform stage — not the raw source IDs — which is standard warehouse design practice so historical fact rows stay stable even if source system IDs change.

## Pipeline Stages

1. **Extract** — simulates pulling raw source data with realistic messiness: duplicate customer rows, null city values, and a handful of negative/garbage transaction amounts.
2. **Transform** — deduplicates customers, fills nulls, resolves natural keys to surrogate keys, filters out invalid transactions (negative amounts, unresolvable foreign keys), and builds the conformed star schema tables.
3. **Load** — writes all five tables into the SQLite warehouse and creates indexes on the fact table's foreign keys for query performance.

Each stage logs row counts in and out, so data loss/filtering is visible and auditable — not silent.

## Project Structure

```
├── etl/
│   └── pipeline.py            # full ETL pipeline (extract/transform/load)
├── sql/
│   └── analysis_queries.sql   # 5 analytical queries against the star schema
├── data/                      # generated by running the pipeline (gitignored)
└── requirements.txt
```

## Running it

```bash
pip install -r requirements.txt
python etl/pipeline.py
```

This regenerates the synthetic data and rebuilds `data/finance_dw.db` from scratch. Then run any query in `sql/analysis_queries.sql` against it, e.g.:

```bash
sqlite3 data/finance_dw.db < sql/analysis_queries.sql
```
